# セキュリティ規約

プロジェクトにおけるバリデーション、認可、秘密情報、およびデータ保護に関するガイドラインです。

## 哲学
- 多層防御、最小権限、デフォルトでの拒否（Deny by Default）
- 境界でのバリデーション、コンテキストに応じたサニタイズ
- 認証（誰か）と認可（何ができるか）を明確に分離する

## 入力と出力
- APIの境界とUIフォームでバリデーションを実施し、制約を強制する
- 許可リスト（Allow-list）を優先し、不正な入力は早期にエラーを返す
- 出力は宛先（HTML, SQL等）に応じて適切にエスケープする

## 認証と認可 (Spring Security)
- **認証**: Spring Securityを使用して身元を確認し、セキュアなセッション/トークンを発行する
- **認可**: アクションを実行する前に、メソッドレベル（`@PreAuthorize`）やURLレベルで権限をチェックする
- 認可ロジックを分散させず、セキュリティ設定クラスで一元管理する

パターン例:
```java
@PreAuthorize("hasRole('ADMIN')")
public void deleteUser(int userId) { ... }
```

## 秘密情報と設定
- パスワード、APIキー等の秘密情報をソースコードにコミットしない
- パスワードは必ずハッシュ化（例: BCrypt）して保存する
- 秘密情報は環境変数またはシークレットマネージャーで管理する

## データの保護
- 削除は原則として `delete_flg` による論理削除を行い、データの追跡性を維持する
- ログにパスワード、トークン、機密性の高い個人情報を出力しない
- 通信は常に TLS (HTTPS) で暗号化する

## セッション管理
- CSRF対策を有効にする
- セッション固定攻撃対策、セキュアなクッキー設定（HttpOnly, Secure）を適用する
- ログアウト時や一定時間経過後にセッションを適切に無効化する

---
_このドキュメントはパターンと原則に焦点を当てています。具体的な設定値は各環境の構成ドキュメントを参照してください。_
