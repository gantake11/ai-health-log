# テスト規約

プロジェクトにおけるテストの対象、配置、および構造に関するガイドラインです。

## 哲学
- 実装ではなく振る舞いをテストする
- 高速で信頼性の高いテストを優先し、壊れやすいモックを最小限に抑える
- 100%のカバレッジを目指すよりも、クリティカルなパスを深くカバーすることを優先する

## 構成
- 配置: `src/test/java` に実装クラスと対応するパッケージ構成で配置する
- 命名: クラス名は `{対象クラス}Tests.java` とする

## テストの種類
- **Unit Test**: 単一のクラスをテスト。依存関係は Mockito でモック化する。高速に実行される。
- **Integration Test**: 複数のコンポーネントの連携をテスト。`@SpringBootTest` を使用し、必要に応じて実際のデータベース（または H2/Testcontainers）を使用する。
- **Security Test**: 認可・認証の挙動をテスト。`spring-security-test` の `@WithMockUser` などを活用する。

## 構造 (Given-When-Then / AAA)
```java
@Test
void doesX_whenY() {
  // Arrange (Given)
  // 入力データの準備、モックの設定
  
  // Act (When)
  // 対象メソッドの実行
  
  // Assert (Then)
  // 結果の検証
  assertThat(result).isEqualTo(expected);
}
```

## モックとデータ
- 外部APIや複雑な依存関係はモック化するが、テスト対象システム自体はモック化しない
- テストデータは最小限にし、テストの意図が明確になるように設定する
- テスト間で状態を共有しないように、実行ごとにリセットする

## ツール
- **JUnit 5**: アサーションやライフサイクル管理
- **Mockito**: モックの作成と検証
- **AssertJ**: 読みやすい流動的なアサーション（`assertThat`）
